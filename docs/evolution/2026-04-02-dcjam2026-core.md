## Feature Summary and Business Context

**Game**: Ember's Escape — a TUI dungeon crawler built for DCJam 2026.
**Premise**: You are Ember, a young dragon descending a 3–5 floor dungeon to reclaim your stolen egg from the human guards left behind by the hero who killed your parent.
**Platform**: macOS/Linux terminal (ANSI-capable, 80×25 fixed layout).
**Language**: Swift 6.3, zero external dependencies.
**Jam scope**: Submittable jam entry. Every feature must be necessary for completion; post-jam expansion is explicitly deferred.

The core design differentiator is Dash as the primary locomotion verb — passing through enemy squares without stopping, consuming a charge, and advancing 3 squares. Combat is a last resort. Brace provides a timed parry window. Special is a charge-meter power move. The game teaches Dash organically by starting the Special charge meter at 0, making Dash the natural first-encounter choice without tutorial text.

## Key Wave Decisions

### DISCOVER Wave (Scout — 2026-04-02)

| Decision | Summary |
|----------|---------|
| DEC-01 | Dash is the primary verb. Combat is opt-in, a last resort. |
| DEC-02 | No timers as default mechanics. Cooldown is readiness state, not countdown pressure. |
| DEC-03 | Two emotional beats explicitly designed: Special = power/badass; Egg + Exit = actual relief. |
| DEC-04 | Dragon vocabulary (claw, fang, wing, lunge, surge) in all combat and movement strings. |
| DEC-05 | Narrative frame: dragon reclaiming stolen egg from hero who killed its parent. |
| DEC-07 | Dash takes priority over "no movement in combat" — combat must permit Dash activation. |
| DEC-08 | Dash: 2 charges, ~45s cooldown per charge, configurable. Upgrades: cooldown reduction + cap increase. |
| DEC-09 | Elemental Rock Paper Scissors explicitly out of scope. |
| DEC-10 | Special available from start, charge meter starts empty — first encounter organically teaches Dash priority. |
| DEC-11 | Floor count: minimum 3, target 5. Floor 1 = starter, Floors 2–4 = regular (egg on one), Floor 5 = boss + exit. |
| DEC-12 | Jam-only scope. Every feature must be necessary for a submittable entry. |

### DISCUSS Wave (Luna — 2026-04-02)

| Decision | Summary |
|----------|---------|
| DISC-01 | Combat action set: Brace + Dash + Special only. No slash/melee attack in this release. |
| DISC-02 | Stat modification via milestone upgrade choices (Option A). Pool of 6–8 upgrades; present 3 at milestone. |
| DISC-03 | Normal movement locked in encounters. Dash exits encounter AND advances 3 squares — always forward, never retreat. |
| DISC-04 | Three-region TUI layout (first-person view / status line / Thoughts panel). Combat on separate dedicated screen. Combat is real-time. |

### DESIGN Wave (Morgan — Solution Architect, 2026-04-02)

| Decision | Summary |
|----------|---------|
| ARCH-01 | No external TUI library. Custom `TUILayer` module (~150–200 lines) using raw ANSI escape codes. |
| ARCH-02 | Value-oriented OOP: domain types as `struct`, pure functions for transformations, `protocol` for port boundaries, plain `class GameLoop`. Confirmed by developer. |
| ARCH-03 | First-person rendering via pre-computed lookup table (~20–25 frames indexed by depth + openings). Zero per-frame floating-point. |
| ARCH-04 | Ports-and-adapters, modular single process: six SwiftPM modules. `GameDomain` has zero imports. |
| ARCH-05 | Synchronous 30 Hz game loop. No async/await in hot path. Delta-time injection for timers. |
| ARCH-06 | `ScreenMode` enum owned by domain: `.dungeon`, `.combat`, `.narrativeOverlay`, `.upgradePrompt`, `.deathState`, `.winState`. |

Critical platform decisions (from CLAUDE.md):
- Terminal write must use a looping write pattern handling short writes and EINTR.
- Non-blocking input via a separate `/dev/tty` fd (`O_RDONLY | O_NONBLOCK`). Never call `fcntl(STDIN_FILENO, F_SETFL, O_NONBLOCK)` — on macOS this makes stdout non-blocking and corrupts multi-byte UTF-8.
- Brace = timed invulnerability window (0.5s), NOT damage reduction. Parry grants 15% Special bonus.

### DISTILL Wave (Quinn — Acceptance Test Designer, 2026-04-02)

| Decision | Summary |

## Lessons Learned

### What Worked Well

1. **Modular walking skeleton first.** The 6-epic roadmap ordered correctly: walking skeleton before Dash Mechanics before Floor Navigation before Combat. Dependencies were resolved naturally. Steps 01-11 and 01-12 correctly blocked on 04-01 and executed later without issue.

2. **Pre-existing implementations accelerated delivery.** A large proportion of steps (particularly in Epics 01 and 02) had `NOT_APPLICABLE` RED_UNIT phases — the implementation pre-existed from the walking skeleton scaffolding and the acceptance tests passed immediately. This is expected and correct in outer-loop TDD: the acceptance test confirms behavior without requiring a separate unit-test gap to close.

3. **Value-oriented architecture made tests trivially fast.** All tests are deterministic, stateless, and inject delta-time directly. No mocks, no terminal, no async. Test suite runs in milliseconds.

4. **Domain isolation enforced by build tool.** SwiftPM module boundaries prevented accidental TUI/IO dependencies in `GameDomain`. Architecture violations are compile errors, not runtime surprises.

5. **Single-file-per-epic test organization.** Six test files (WalkingSkeletonTests, DashMechanicsTests, FloorNavigationTests, CombatTests, ProgressionTests, WinLossConditionTests) map 1:1 to roadmap epics. Easy navigation for future maintainers.

### Issues Encountered

1. **Step re-ordering mid-delivery.** Steps 01-07 through 01-12 were held back and executed in a second batch after combat dependency resolution. The execution log reflects the actual execution order, not roadmap order. Future deliveries should front-load dependency analysis more explicitly in the roadmap step ordering before execution begins.

2. **Staircase visibility gap (known, logged to MEMORY.md).** The depth=0 frame was never shown to the player because the staircase tile is at depth=0 when the player steps onto it — the floor transition fires immediately. The player never sees the staircase frame. Planned fix: introduce a `NarrativeEvent.stairsDescend` overlay (similar to egg discovery) that pauses on the staircase frame before advancing floors.

3. **Combat epic intentionally minimal.** Only 3 of 13 Combat steps executed (playtest-minimal scope). The 10 deferred steps cover Brace detailed mechanics, Special full cycle, boss defeat, and parry-to-Special bonus. These are the most complex interactions and were correctly deferred to avoid over-building before playtesting validates the core loop.

4. **Non-blocking input gotcha (critical platform knowledge).** The initial approach of `fcntl(STDIN_FILENO, F_SETFL, O_NONBLOCK)` would have made stdout non-blocking on macOS (shared tty file description), corrupting UTF-8 multi-byte sequences mid-frame. Resolution: separate `/dev/tty` fd for input. This was validated during the DESIGN wave and codified in CLAUDE.md.
